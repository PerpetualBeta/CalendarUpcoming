# CalendarUpcoming

A macOS menu-bar app that monitors your calendars for upcoming events and alerts you with a pulsing blue icon.

## What it does

- Sits silently in the menu bar as a plain calendar icon
- When an event is starting within your configured look-ahead window, the icon switches to blue and slowly pulses with a glow
- **Left-click** ‚Äî opens a popover listing all upcoming events: time until start, clock time, and calendar name with colour bar
- **Right-click** ‚Üí **Settings** to change the look-ahead period, or **Quit**
- **ESC or click outside** ‚Äî closes the popover

Monitors all calendars registered with macOS, including Apple Calendar, Outlook/Exchange (via System Settings ‚Üí Internet Accounts), Google, and iCloud.

## Building

Requires Xcode command-line tools and a valid Apple developer certificate.

```bash
xcodebuild -project CalendarUpcoming.xcodeproj \
  -target CalendarUpcoming \
  -configuration Release \
  CONFIGURATION_BUILD_DIR=/tmp/CalendarUpcoming_build \
  CODE_SIGN_STYLE=Manual \
  CODE_SIGN_IDENTITY="Apple Development: yourname@example.com (TEAMID)" \
  build

cp -r /tmp/CalendarUpcoming_build/CalendarUpcoming.app ~/Desktop/
```

> **Note:** The app must be code-signed. macOS TCC (the permissions system) uses the code signature to track the calendar access grant ‚Äî an unsigned build will show "access denied" even after granting permission in System Settings.

On first launch, macOS will ask for calendar permission. Grant **Full Access**. If the popover shows "Full calendar access needed", open **System Settings ‚Üí Privacy & Security ‚Üí Calendars**, find CalendarUpcoming, and switch it from *Write Only* to *Full Access*.

## Architecture

| File | Purpose |
|---|---|
| `CalendarUpcomingApp.swift` | `@main` entry; wires `AppDelegate` via `@NSApplicationDelegateAdaptor` |
| `AppDelegate.swift` | Owns the `NSStatusItem`, manages the pulse animation and popover lifecycle |
| `EventMonitor.swift` | `ObservableObject` wrapping `EKEventStore`; polls every 30s and on `EKEventStoreChanged` |
| `EventsPopoverView.swift` | SwiftUI popover content ‚Äî event list, empty state, access-denied states |
| `SettingsView.swift` | Look-ahead picker (5 / 10 / 15 / 30 / 60 min), persisted to `UserDefaults` |

### EventMonitor

- Holds a single `EKEventStore` instance (never recreated)
- On macOS 14+: calls `requestFullAccessToEvents`; on macOS 13: calls `requestAccess(to: .event)`
- Checks `EKEventStore.authorizationStatus` directly on startup to avoid redundant permission dialogs
- Detects **write-only** access (macOS 14+ split calendar permission into two tiers) and surfaces a specific message in the popover
- Filters out all-day events and events the user has declined
- `@Published upcomingEvents: [EKEvent]` ‚Äî AppDelegate subscribes via Combine to drive icon state

### AppDelegate

- `NSApp.setActivationPolicy(.accessory)` ‚Äî hides from Dock
- **Pulse animation:** `CABasicAnimation` on `opacity` (1.0 ‚Üí 0.35) and `shadowOpacity` (0.9 ‚Üí 0.1), both 1.2s easeInEaseOut, autoreversing, infinite
- **Icon:** idle = template `calendar` symbol (adapts to light/dark); active = `calendar.badge.exclamationmark` with all-palette-layers forced blue via `NSImage.SymbolConfiguration(paletteColors:)`
- **Outside-click dismissal:** uses `NSEvent.addGlobalMonitorForEvents` (required for accessory-mode apps ‚Äî `.transient` popover behaviour doesn't receive outside clicks)
- **Right-click menu:** assign menu to `statusItem.menu`, call `performClick`, then nil it out ‚Äî standard AppKit pattern for menu-bar apps

### Icon

Generated by `generate_icon.swift` (run with `swift generate_icon.swift <output-dir>`). Draws using Core Graphics: blue gradient rounded-rect background, white calendar body with header and binding rings, a day-grid, and a clock badge in the bottom-right corner. Outputs all 10 required PNG sizes into `Assets.xcassets/AppIcon.appiconset/`.
